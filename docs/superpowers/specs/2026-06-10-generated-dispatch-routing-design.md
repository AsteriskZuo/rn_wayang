# Generated Dispatch Routing Design

## Purpose

Prepare `measured_app` for SDK API alignment and future full JMeter coverage by generating the mechanical dispatch routing layer from the local `react-native-chat-sdk` type declarations and the existing handwritten `Biz*` wrappers.

The current `Dispatch.ts` contains a large handwritten switch. Most cases only map:

```ts
case 'selectPushTemplate': {
  BizChatPushManager.selectPushTemplate(info, callback);
  break;
}
```

That structure is repetitive, easy to drift during SDK version changes, and expensive to maintain by hand. The generated route layer should remove this mechanical maintenance without generating the reliability-sensitive `Biz*` wrapper implementations.

## Core Decision

Generate only the manager-specific dispatch routing files.

Do not generate `Biz*` wrappers. The wrapper layer is intentionally handwritten because each API can require different `info` extraction, field aliases, enum normalization, message construction, prefetching, callback handling, or SDK-specific behavior. Generating that logic would reduce reliability.

`Dispatch.ts` remains the handwritten protocol entry point. Its public method stays:

```ts
dispatch(data: any, callback: ReturnCallback): boolean
```

The generated files live under:

```text
measured_app/src/dispatch/
```

## Source Of Truth

The generator uses two local source sets:

1. SDK type declarations:

```text
measured_app/node_modules/react-native-chat-sdk/lib/typescript/src/*.d.ts
```

2. Existing handwritten wrappers:

```text
measured_app/src/biz/Biz*.ts
```

Target SDK APIs are public manager methods that:

- are declared on one of these SDK classes:
  - `ChatClient`
  - `ChatManager`
  - `ChatGroupManager`
  - `ChatRoomManager`
  - `ChatContactManager`
  - `ChatPresenceManager`
  - `ChatPushManager`
  - `ChatUserInfoManager`
- return `Promise<...>`;
- are not marked with `@deprecated`.

The generator only emits a route when the matching `Biz*` class already has a static method with the same name.

## Generated File Layout

The generated route layer uses one file per SDK manager plus a barrel export:

```text
measured_app/src/dispatch/
  ChatClient.generated.ts
  ChatManager.generated.ts
  ChatGroupManager.generated.ts
  ChatRoomManager.generated.ts
  ChatContactManager.generated.ts
  ChatPresenceManager.generated.ts
  ChatPushManager.generated.ts
  ChatUserInfoManager.generated.ts
  Internal.ts
  index.ts
```

`Internal.ts` may be handwritten because it covers protocol helper commands that are not part of SDK API coverage.

`index.ts` exports every manager route function:

```ts
export {dispatchChatClient} from './ChatClient.generated';
export {dispatchChatManager} from './ChatManager.generated';
export {dispatchChatGroupManager} from './ChatGroupManager.generated';
export {dispatchChatRoomManager} from './ChatRoomManager.generated';
export {dispatchChatContactManager} from './ChatContactManager.generated';
export {dispatchChatPresenceManager} from './ChatPresenceManager.generated';
export {dispatchChatPushManager} from './ChatPushManager.generated';
export {dispatchChatUserInfoManager} from './ChatUserInfoManager.generated';
export {dispatchInternal} from './Internal';
```

## Dispatch Entry Point

`measured_app/src/Dispatch.ts` remains handwritten and keeps protocol-level behavior:

1. reject empty data;
2. parse JSON;
3. log parsed `cmd` and `info`;
4. call generated manager dispatchers in a fixed order;
5. call internal helper dispatch last;
6. return `false` only when no route handles the command.

The dispatch body should be shaped like this:

```ts
return (
  dispatchChatClient(cmd, info, callback) ||
  dispatchChatManager(cmd, info, callback) ||
  dispatchChatGroupManager(cmd, info, callback) ||
  dispatchChatRoomManager(cmd, info, callback) ||
  dispatchChatContactManager(cmd, info, callback) ||
  dispatchChatPresenceManager(cmd, info, callback) ||
  dispatchChatPushManager(cmd, info, callback) ||
  dispatchChatUserInfoManager(cmd, info, callback) ||
  dispatchInternal(cmd, info, callback)
);
```

## Generated Route Shape

Each generated manager file exports one function.

Example:

```ts
import {ReturnCallback} from '../RNWS';
import {BizChatPushManager} from '../biz/BizChatPushManager';

export function dispatchChatPushManager(
  cmd: string,
  info: any,
  callback: ReturnCallback,
): boolean {
  switch (cmd) {
    case 'selectPushTemplate':
      BizChatPushManager.selectPushTemplate(info, callback);
      return true;
    case 'fetchSelectedPushTemplate':
      BizChatPushManager.fetchSelectedPushTemplate(info, callback);
      return true;
    default:
      return false;
  }
}
```

Generated files should contain a header that makes their source clear:

```ts
// Generated by scripts/generate-dispatch-routes.js.
// Review generated diffs before committing.
```

Generated files are committed as ordinary source. They are not runtime dynamic routing.

## Generator Script

Create a generator script:

```text
measured_app/scripts/generate-dispatch-routes.js
```

The script should use the TypeScript compiler API to parse TypeScript declarations and source files. It should not depend on fragile line-oriented regex parsing for method extraction.

The script responsibilities are:

1. read the configured SDK manager `.d.ts` files;
2. extract non-deprecated Promise-returning methods;
3. read the configured `Biz*` wrapper files;
4. extract static methods from each `Biz*` class;
5. generate route cases only for methods present in both SDK active APIs and the matching `Biz*` class;
6. write `src/dispatch/*.generated.ts`;
7. write `src/dispatch/index.ts`;
8. print a generation summary.

Add a package script:

```json
"generate:dispatch": "node scripts/generate-dispatch-routes.js"
```

## Missing Wrapper Rule

If an active SDK API exists but the corresponding `Biz*` static wrapper does not exist, the generator must not create a dispatch case for it.

Missing wrappers are reported, not implemented by the generator:

```text
missing wrapper, not generated:
- ChatManager.searchMessages
- ChatManager.searchMessagesInConversation
```

This report is an input to later wrapper implementation work. It is not a generator failure by itself.

After the missing `Biz*` wrapper is implemented manually, rerunning the generator should include the API in the generated route file.

## Deprecated API Rule

Deprecated SDK APIs are not generated as normal SDK routes.

The generator should report deprecated SDK methods that have same-named `Biz*` wrappers so they can be reviewed:

```text
deprecated SDK API skipped:
- ChatClient.login
- ChatRoomManager.joinChatRoom
```

This report does not automatically delete the wrapper. It only prevents deprecated SDK methods from entering the generated SDK route layer.

## Protocol Exceptions

Some commands are not normal active SDK coverage but still matter for the test tool.

These must be handled outside generated SDK route files, preferably in `measured_app/src/dispatch/Internal.ts`:

- `init`
- `login`
- `addConnectionDelegate`
- `deleteConnectionDelegate`
- `addMultiDeviceDelegate`
- `deleteMultiDeviceDelegate`
- `addContactManagerDelegate`
- `removeContactManagerDelegate`
- `addChatManagerDelegate`
- `removeChatManagerDelegate`
- `addRoomManagerDelegate`
- `removeRoomManagerDelegate`
- `addGroupManagerDelegate`
- `removeGroupManagerDelegate`

`login` remains a test setup exception because JMeter currently needs username/password login. It is not counted as active SDK API coverage.

`init` is protocol setup and remains explicit.

Listener/delegate helpers are internal tool helpers and should not be mixed into generated SDK coverage routes.

## SDK Version Changes

The generator is a maintenance aid, not a source of unquestioned truth.

For SDK upgrade or downgrade work:

1. update the SDK dependency;
2. run the generator when route refresh is useful;
3. review the generated diff;
4. inspect the summary for missing wrappers and deprecated skips;
5. manually adjust `Biz*` wrappers when needed;
6. rerun the generator after wrapper changes;
7. validate and use git commits/tags to mark the confirmed state.

The repository state, review diff, and git tags remain the final version-management mechanism.

## Validation

The generated routing task should include scriptable validation:

1. generated route files contain no duplicate command cases;
2. every generated command calls an existing static method on the matching `Biz*` class;
3. generated route files do not include deprecated SDK APIs;
4. `Dispatch.ts` imports and calls the generated route functions;
5. `Internal.ts` contains only explicit protocol exceptions and helper commands;
6. the generator summary reports missing wrappers without treating them as generated routes.

Run app static validation from `measured_app`:

```bash
yarn generate:dispatch
yarn lint
yarn test
```

No mocked `Dispatch.ts` route tests are required for this task. `measured_app` is the test puppet/tool itself, so behavior issues should surface through generated route validation, static checks, and later real relay/JMeter execution.

## Completion Criteria

This task is complete when:

- old handwritten SDK route cases are removed from `Dispatch.ts`;
- `Dispatch.ts` remains the handwritten protocol entry point;
- generated manager route files exist under `measured_app/src/dispatch/`;
- `src/dispatch/index.ts` exports generated manager dispatchers;
- protocol exceptions are isolated outside generated SDK routes;
- the generator can be rerun and produces stable output;
- missing wrappers are clearly reported;
- deprecated SDK APIs are not generated as normal SDK route cases;
- `yarn generate:dispatch`, `yarn lint`, and `yarn test` pass in `measured_app`.
